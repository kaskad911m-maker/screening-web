## Что это за проект (одним абзацем)

- **Сайт и мини-приложение** — обычные HTML-файлы (`index.html`, `mini-app.html` и др.). Их открывает браузер или Telegram.
- **Backend** — маленькая программа на Node.js, которая **принимает заявки** и кладёт их в **базу SQLite** на диске (`backend/data/app.db`).
- **Supabase** (домашка шаг 1) — те же таблицы **в интернете**; приложение к ним подключится позже, когда дойдёшь до следующего шага урока. Сейчас для работы дома достаточно SQLite.

## Часть 1. Запуск у себя на компьютере

### Шаг 1. Установить Node.js
Скачай с сайта **https://nodejs.org** (LTS). Установи как обычную программу.

### Шаг 2. Открыть папку backend
В проводнике зайди в проект → папка **`backend`**.

### Шаг 3. Один раз установить зависимости
Открой терминал в этой папке и выполни:

```text
npm install
```

### Шаг 4. Создать файл настроек
В папке `backend` должен быть файл **`.env`**. Проще всего скопировать пример:

**Windows (PowerShell), находясь в папке `backend`:**

```text
Copy-Item .env.example .env
```

В `.env` можно оставить как есть для учёбы: порт **8787** и токен **dev-admin-token**.

### Шаг 5. Запустить сервер

```text
npm start
```

Окно **не закрывай** — пока оно открыто, заявки принимаются.

### Шаг 6. Проверить, что сервер жив
В браузере открой:

```text
http://localhost:8787/api/health
```

Должно показать что-то вроде: `ok: true`.

### Шаг 7. Открыть сайт и мини-приложение не как «файл с диска»
Из **корня проекта** (где лежит `index.html`) в терминале:

```text
npx serve .
```

Браузер покажет адрес (часто `http://localhost:3000`). Оттуда открывай страницы.

**Мини-приложение с связью с сервером:**

```text
http://localhost:3000/mini-app.html?api=http://localhost:8787
```

**Админка (список записей из Mini App):**

```text
http://localhost:3000/admin.html
```

(Ключ по умолчанию совпадает с `ADMIN_TOKEN` в `.env`.)

## Часть 2. Vercel — связать эту папку с проектом на vercel.com

**Что получится:** по адресу вида `https://…vercel.app` откроются `index.html`, `mini-app.html` и остальные файлы из **корня** проекта. Папка **`backend/`** в деплой **не попадает** (см. `.vercelignore`) — API для заявок нужно держать отдельно (VPS, Railway и т.д.) или указывать уже существующий **https**-URL в `mini-app.html` (meta `booking-api-base` или `?api=` в ссылке Mini App).

### Вариант А — как раньше: GitHub + автодеплой (удобнее всего)

1. Залей проект на **GitHub** (репозиторий может быть приватным).
2. Зайди на **https://vercel.com** → войди тем же аккаунтом, что и для проекта «веб-сайт для скрининга».
3. **Add New… → Project** → **Import** свой репозиторий с `my-first-project`.
4. Настройки сборки:
   - **Framework Preset:** Other  
   - **Root Directory:** `./` (корень)  
   - **Build Command:** оставить пустым  
   - **Output Directory:** оставить пустым или `.`
5. **Deploy.** После этого каждый `git push` в выбранную ветку (например `main`) будет обновлять сайт.

Если проект **уже есть** (например `screening-web`): **Project → Settings → Git** → подключи репозиторий или смени репозиторий на актуальный.

### Вариант Б — без Git: Vercel CLI с компьютера

1. Установи CLI один раз (в PowerShell):

```text
npm install -g vercel
```

2. Войди в аккаунт:

```text
vercel login
```

3. Перейди в **корень** проекта (где лежит `index.html`) и привяжи папку к существующему проекту или создай новый:

```text
cd путь\к\my-first-project
vercel link
```

В вопросах выбери **team**, потом пункт **Link to existing project** и найди свой проект (например **screening-web**), либо **Create new project**.

4. Пробный деплой:

```text
vercel
```

5. Продакшен:

```text
vercel --prod
```

После этого в папке появится каталог **`.vercel`** (куда Vercel сохранил `projectId` и `orgId`) — его **не удаляй**, если хочешь повторять `vercel --prod` из этой же папки.

### Ссылка для Telegram Mini App

В настройках бота укажи URL вида:

```text
https://твой-проект.vercel.app/mini-app.html
```

Если API не на том же домене, добавь параметр (подставь свой https-адрес API):

```text
https://твой-проект.vercel.app/mini-app.html?api=https://твой-api.example.com
```

## Часть 2б. GitHub Pages

Если фронт уже на **GitHub** (репозиторий `screening-web` или другой):

1. **GitHub** → твой репозиторий → **Settings** → **Pages**.  
2. **Source:** ветка **main**, папка **/ (root)** → **Save**.  
3. Сайт будет по адресу вида: `https://ИМЯ.github.io/ИМЯ-РЕПО/`  
4. **Лендинг:** `…/index.html` или корень `/`.  
5. **Mini App для бота:** `…/mini-app.html` — именно этот URL укажи в **BotFather** → Web App.  
6. **Важно:** на GitHub попадает только статика. **Backend** там не крутится — заявки с формы и Mini App должны уходить на **отдельный https-API** (VPS, Railway и т.д.). В `index.html` и `mini-app.html` в `<meta name="booking-api-base" content="…">` или в ссылке `mini-app.html?api=…` укажи этот адрес.

После каждого `git push` подожди 1–2 минуты и при необходимости открой страницу с `?v=2` в конце URL, чтобы сбросить кэш браузера.

## Часть 3. Supabase (домашнее задание, шаг 1)

Подробно и по кликам: файл **`SUPABASE-ШАГ1.md`**.  
Коротко: зайти на **supabase.com** → создать проект → **SQL Editor** → вставить **`supabase/schema.sql`** → **Run** → проверить в **Table Editor** таблицы **masters**, **bookings**, **leads**.

Ключи **URL** и **anon** лежат в **Project Settings → API** — пригодятся, когда будешь подключать API к облаку (следующие шаги курса).

## Часть 4. Когда захочешь выложить в интернет (общая схема)

1. **Статические страницы** (`index.html`, `mini-app.html`, …) — удобно на **Vercel**, **Netlify** или любом хостинге с HTML.
2. **Backend** с SQLite — лучше на **VPS** (например, Beget Cloud, Ubuntu), потому что программа должна работать постоянно и писать на диск.
3. На сервере: установить Node.js, скопировать проект, в `backend` сделать `npm install`, настроить `.env` (**свой** `ADMIN_TOKEN`!), запустить через **PM2** или systemd, открыть порт или поставить **Nginx** как прокси.
4. В **`mini-app.html`** и **`index.html`** в теге `<meta name="booking-api-base" content="...">` указать **https**-адрес твоего API. В Telegram Mini App адрес страницы тоже https — и API должен быть https, иначе браузер может заблокировать запросы.

Откат: вернуть старый рабочий код и перезапустить процесс на сервере.

## Чек-лист перед публикацией

Сверься по пунктам **до** того как отдавать ссылку клиенту или куратору:

- [ ] В **BotFather** у мини-приложения указан полный URL на **`/mini-app.html`**, а не на корень сайта.  
- [ ] В продакшене в HTML указан **https**-адрес API (`booking-api-base` или `?api=`).  
- [ ] Форма на `index.html` протестирована с **реальным** backend (заявка видна в админке или в `GET /api/leads`).  
- [ ] В `.env` на сервере **свой** `ADMIN_TOKEN`, не учебный из примера.  
- [ ] В репозиторий **не** попали секреты (токены ботов, ключи Supabase **service role**).  
- [ ] (Опционально) В Supabase выполнен `schema.sql`, таблицы **masters**, **bookings**, **leads** на месте — см. `SUPABASE-ШАГ1.md`.

## Переменные окружения (backend `.env`)

| Переменная | Зачем |
|------------|--------|
| `PORT` | Порт HTTP-сервера (локально часто 8080 или 8787). |
| `ADMIN_TOKEN` | Секрет для `admin.html` и заголовка `x-admin-token` в API. |
| `BOOKING_NOTIFY_BOT_TOKEN` | Токен бота для уведомлений в Telegram о новых заявках (можно пусто). |
| `BOOKING_NOTIFY_CHAT_ID` | ID чата/группы для уведомлений (можно пусто). |
| `SUPABASE_URL` | После подключения облака — URL проекта (из Supabase → API). |
| `SUPABASE_ANON_KEY` | Публичный ключ клиента (когда фронт ходит в Supabase напрямую). |
| `SUPABASE_SERVICE_ROLE_KEY` | Только на сервере, никогда во фронт; для обхода RLS при записи с backend. |

Шаблон с комментариями: **`backend/.env.example`**.

## Часть 5. Чек-лист «всё работает»

1. `GET /api/health` → `ok: true`
2. Из `mini-app.html` с параметром `?api=...` нажать «Записаться» → экран «Спасибо»
3. В `admin.html` видна новая строка (или `GET /api/bookings` с заголовком `x-admin-token`)
4. Форма на `index.html` отправляется без ошибки (заявка в `leads` в SQLite)
5. В Supabase в Table Editor есть таблицы после `schema.sql`
